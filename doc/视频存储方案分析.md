# 视频存储方案分析

## 1. 当前项目现状

本轮优化前，视频任务结果只写入了占位 URL，并没有形成真正的“生成后落库、可访问、可替换”的存储链路。

现在已经补齐为：

1. 视频生成服务先拿到上游返回的视频 URL
2. 再把视频内容转成字节流
3. 通过可配置存储后端落地
4. 将最终可访问地址写入 `video_tasks.video_url`

当前代码已支持的后端：

- `local`
- `github_release`
- `ipfs`
- `s3_compatible`

对应实现位于：

- `src/app/services/storage.py`

## 2. 各方案可行性判断

### 2.1 local

适合：

- 本地开发
- MVP 联调
- 比赛演示
- 小规模单机部署

优点：

- 接入最简单
- 零额外平台成本
- 访问路径清晰
- 调试体验最好

缺点：

- 不适合多实例部署
- 不适合跨机器共享
- 不适合作为长期正式视频库

结论：

`最适合作为当前项目默认方案。`

### 2.2 GitHub

这里需要分成两种方式看。

#### 2.2.1 直接放在 Git 仓库

不建议。

原因：

- Git 不适合频繁写入大体积二进制视频
- 仓库体积会快速膨胀
- clone / pull 成本会越来越高
- 后续清理和版本治理都不顺手

#### 2.2.2 放在 GitHub Release 资产

技术上可行，项目也已经预留了 `github_release` 存储适配。

适合：

- 少量演示样片
- 样例文件外链
- 比赛汇报时的固定素材

不适合：

- 高频视频生成
- 批量审核素材
- 需要覆盖更新、清理过期、权限分级的正式业务

结论：

`可以用，但更像演示资产分发，不适合做当前项目主视频库。`

### 2.3 IPFS

技术上可接入，项目也已经预留了 `ipfs` 适配。

但是 IPFS 有一个常见误区：

`把文件 add 到 IPFS，不等于你已经获得稳定、长期、免费的在线视频存储。`

如果只是简单 add：

- 不等于会永久保存
- 不等于访问一定稳定
- 不等于网关速度一定可控

真正要稳定使用，通常还要配合：

- 自建节点
- pinning service
- 稳定网关

适合：

- 技术展示
- 去中心化内容分发实验
- 对内容寻址有明确需求的场景

不适合：

- 审核链路主存储
- 追求稳定回放和稳定下载的后台系统

结论：

`可以接，但不建议作为当前项目的主存储方案。`

### 2.4 S3 兼容对象存储

这是本轮新增的更实用方案，适用于：

- MinIO
- AWS S3
- Cloudflare R2
- 阿里云 OSS 的 S3 兼容模式
- 腾讯云 COS 的 S3 兼容模式

优点：

- 原生适配二进制媒体对象
- 便于做生命周期管理
- 便于多实例部署
- 更适合后续接 CDN
- 更适合做权限、访问域名、成本核算

缺点：

- 需要额外平台配置
- 正式环境通常不是“完全免费”

结论：

`如果后续要从比赛原型走向正式化，这是最合适的迁移方向。`

## 3. 当前项目最推荐的落地路线

### 阶段一：当前开发 / 比赛演示

优先顺序：

1. `local`
2. 如需公网外链小样片，再用 `github_release`

原因：

- 上手快
- 成本最低
- 调试最稳

### 阶段二：后续正式部署

优先顺序：

1. `s3_compatible`
2. 配合 CDN

原因：

- 更适合视频文件
- 更适合多实例
- 更适合生命周期和权限管理

## 4. 关于“免费存储”的现实建议

如果问题是：

`有没有既免费、又稳定、又适合长期存视频的方案？`

现实答案通常是：

`几乎没有真正适合正式业务的长期免费方案。`

看起来“免费”的方案，通常会在这些点上出现约束：

- 空间限制
- 带宽限制
- 访问不稳定
- 鉴权能力弱
- 生命周期治理差

所以更务实的判断是：

- GitHub 可用于少量演示素材
- IPFS 可用于技术展示或实验
- 真正稳定的主存储，还是对象存储更合适

## 6. 操作说明

### 6.1 local

`.env` 示例：

```env
VIDEO_STORAGE_BACKEND=local
MEDIA_ROOT=media
MEDIA_URL_PREFIX=/media
MEDIA_BASE_URL=
```

### 6.2 GitHub Release

```env
VIDEO_STORAGE_BACKEND=github_release
GITHUB_STORAGE_OWNER=
GITHUB_STORAGE_REPO=
GITHUB_STORAGE_TOKEN=
GITHUB_STORAGE_RELEASE_TAG=video-assets
```

### 6.3 IPFS

```env
VIDEO_STORAGE_BACKEND=ipfs
IPFS_API_URL=http://127.0.0.1:5001
IPFS_GATEWAY_URL=https://ipfs.io/ipfs
IPFS_PIN_ON_ADD=true
```

### 6.4 S3 兼容对象存储

```env
VIDEO_STORAGE_BACKEND=s3_compatible
S3_ENDPOINT_URL=
S3_ACCESS_KEY_ID=
S3_SECRET_ACCESS_KEY=
S3_BUCKET=
S3_REGION=
S3_OBJECT_PREFIX=videos
S3_PUBLIC_BASE_URL=
```

说明：

- `S3_PUBLIC_BASE_URL` 建议填 CDN 域名或对象存储公网访问域名
- 如果不填，系统会按常见 S3 URL 规则生成回写地址

### 6.5 验证当前后端

启动服务后可访问：

```text
GET /api/v1/operations/storage
```

用来确认当前启用的是哪种存储后端，以及配置是否完整。

## 7. 一句话结论

`GitHub 和 IPFS 都能接，但都不适合作为当前项目的视频主存储；当前最务实的是 local，后续最合理的是 s3_compatible。`
