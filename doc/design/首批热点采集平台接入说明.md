# 首批热点采集平台接入说明

## 1. 目标

当前系统的首批热点采集范围先收敛为 3 个平台：

- `douyin`
- `xiaohongshu`
- `xigua`

这样做的目的不是永久排他，而是先把热点采集、爆款分析、二创脚本生成这前三段流程跑通。后续新增站点时，只需要继续扩展 `CSO/services/hotspot_providers/` 与对应配置，不再改动主流程骨架。

补充约定：

- `bilibili` 能力可以保留在代码中，作为后续可启用平台
- 但当前默认启用列表仍然只包含 `douyin / xiaohongshu / xigua`
- 也就是说，`bilibili` 是“保留支持、暂不列为首批默认采集目标”

## 2. 平台清单

### 2.1 抖音 `douyin`

- 官方主站: [https://www.douyin.com/](https://www.douyin.com/)
- 热门视频入口: [https://www.douyin.com/shipin/](https://www.douyin.com/shipin/)
- 归类: 短视频主阵地
- 当前建议采集方向:
  - 热门视频列表
  - 指定关键词搜索结果
  - 垂类标签页内容
- 接入注意:
  - Web 端存在登录态、风控和动态渲染问题，真实采集建议走浏览器自动化
  - 首期应先保证“能抓取页面中的标题、作者、链接、热度指标”
  - 不在首批阶段处理评论深翻、账号画像深抓、下载原视频等高风险动作

### 2.2 小红书 `xiaohongshu`

- 官方主站: [https://www.xiaohongshu.com/](https://www.xiaohongshu.com/)
- 创作服务平台: [https://creator.xiaohongshu.com/](https://creator.xiaohongshu.com/)
- 归类: 种草/生活方式内容阵地
- 当前建议采集方向:
  - 搜索结果页的高互动笔记
  - 垂类主题页内容
  - 与目标业务领域相关的高赞图文/视频笔记
- 接入注意:
  - Web 端同样存在登录态、风控和动态接口问题
  - 真实采集优先使用浏览器自动化提取标题、作者、封面、互动数、跳转链接
  - 首批不处理私域数据、用户关系链、评论区深度采样

### 2.3 西瓜视频 `xigua`

- 官方主站: [https://www.ixigua.com/](https://www.ixigua.com/)
- 移动 Web: [https://m.ixigua.com/](https://m.ixigua.com/)
- 创作平台: [https://studio.ixigua.com/](https://studio.ixigua.com/)
- 归类: 中长视频/知识视频内容阵地
- 当前建议采集方向:
  - 首页推荐和关键词结果中的高热视频
  - 适合知识型脚本拆解的讲解类、经验类、教程类视频
- 接入注意:
  - 西瓜与抖音内容生态存在交叉，但在内容形态和时长结构上不同，不能简单复用短视频评分规则
  - 首批采集先统一输出热点基础字段，不额外引入“横屏专属复杂规则”
  - 后续如果进入中视频收益或创作者规则对接，再单独扩展平台策略

## 3. 首批统一字段

首批三个站点进入系统后，统一落为 `HotspotItem` 基础结构：

- `platform`
- `content_id`
- `title`
- `author`
- `author_id`
- `url`
- `cover_image`
- `view_count`
- `like_count`
- `comment_count`
- `share_count`
- `category`
- `tags`
- `duration`
- `fetched_at`

首批阶段只要求三个平台都能稳定提供这组基础字段。更深的平台特性字段，不进入主模型，不污染前三段流程。

## 4. 架构归属

- `CSO`
  - 负责热点发现、平台采集适配、候选池构建
  - `hotspot.yaml` 维护首批启用平台列表
  - `services/hotspot_providers/` 维护平台 provider
- `CIO`
  - 负责热点数据结构、持久化、统一 schema
- `CCO`
  - 负责基于热点生成结构化《爆款 DNA 报告》
- `COO`
  - 负责基于分析结果生成脚本，不直接感知平台采集细节

这意味着平台扩展是 `CSO` 主责，不应把平台采集逻辑散落到 `CEO` 或前台路由层。

## 5. 当前落地原则

1. 首批默认启用平台切换为 `douyin / xiaohongshu / xigua`
2. 前三段流程先允许使用 mock provider 跑通端到端结构
3. 真实采集能力单独在 provider 层增强，不改 API 合同
4. `bilibili` 可以保留为待启用平台，但不进入当前默认启用名单
5. 后续新增平台时，先补文档，再补 provider，再补测试
