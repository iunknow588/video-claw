# 体系结构与流程划分总说明

## 1. 文档目的

这份文档给出当前版本的最终结构判断规则，解决四个核心问题：

1. 顶层部门是否已经足够承载现有功能
2. CEO 是否还暴露在第一线
3. 外部接口、内部查询、主流程编排分别由谁负责
4. 后续新增代码应该落到哪里

当前版本不保留兼容层，不再沿用历史命名习惯，所有新代码必须直接落入清晰、单一、稳定的职责边界。

## 2. 顶层部门结论

`src/app/` 当前保留并继续使用的顶层部门为：

```text
CAO  CCO  CEO  CFO  CHO  CIO  CMO  COO  CQO  CSO  CTO
```

结论如下：

- 不再新增新的“路由部门”或“生产部门”来承载现有功能。
- 原先设想的 8 部门结构不足以清晰承接当前能力，至少还需要保留 `COO`、`CSO`、`CCO` 三个独立业务部门。
- `Production` 不单独设新部门，也不归 CTO；生产执行统一归 `COO`。
- `CSO / CCO` 必须保持独立，不能随意并入 CTO、CIO 或 COO。

## 3. 各部门职责

- `CEO`
  只负责公司级治理内核，包括控制平面、组织治理、编排规则、Leader 管理与演进策略。

- `CAO`
  负责所有对外流程入口与行政网关，包括公开控制台、工作流提交入口、治理查询网关、交付结果出口。

- `CMO`
  负责所有对话式入口与用户沟通事件流，包括聊天、进度播报、结果汇报。

- `CIO`
  负责内部数据层、持久化、事件、日志、观测、存储、内部运行查询接口。

- `CFO`
  负责预算闸门、成本估算、资源约束、费用核算。

- `CSO`
  负责热点、趋势、研究线索、素材情报输入。

- `CCO`
  负责内容分析、热点拆解、结构化洞察沉淀。

- `CTO`
  负责技术策划、提示词体系、标题候选、模型适配、生产前技术规范。

- `COO`
  负责脚本、素材、配音、字幕、渲染、视频制作等生产执行链路。

- `CQO`
  负责质量检查、质检报告、返工建议与回退策略。

- `CHO`
  负责公共 Agent 管理、能力目录、健康状态与共享能力治理。

## 4. CEO 隐身原则

CEO 不应该直接面对用户，也不应该作为公开 API 的品牌或路径前缀出现。

当前结构必须遵守下面三条硬规则：

1. 不提供 `/api/ceo/*` 这类公开接口。
2. 不提供 `/ceo` 这类公开页面别名。
3. CEO 只作为内部治理内核被 `CAO / CMO / CIO` 等部门委托调用。

也就是说：

- 用户看到的是 `CAO` 的流程入口和控制台；
- 用户对话接触的是 `CMO`；
- 内部运行查询由 `CIO` 暴露；
- CEO 只在后台决策、编排、治理，不直接接单、不直接展示、不直接对话。

附加说明：内部路由的注册与暴露必须由 `CEO` 内部的协调器负责（例如 `departments.CEO.coordinator`）。外部组件不得直接挂载或修改 `CEO` 的内部路由；若需要跨部门协调或审计，必须通过 `CEO` 协调器提供的受控接口进行。

实现细节：协调器支持基于路由 `tags` 的内部路由标记（示例标签：`control`、`orchestration`、`operations`）。被标记为内部的路由不会出现在 OpenAPI 文档中（`include_in_schema=False`），以确保治理接口不被误用或误暴露。请在新增内部治理路由时使用这些标签或与 `CEO` 团队协调新增标签。

## 5. 主流程划分

当前默认主生产链路为：

```text
CFO -> CSO -> CCO -> CTO -> COO -> CQO -> CAO
```

职责顺序如下：

- `CFO`：先做预算与资源闸门
- `CSO`：生成热点与研究输入
- `CCO`：形成结构化内容分析
- `CTO`：整理成可生产的技术输入
- `COO`：执行脚本、配音、字幕、渲染、成片
- `CQO`：做质量判定与返工建议
- `CAO`：承接交付、发布、对外结果呈现

`CEO` 不在主链路中作为公开阶段出现，只在后台控制主链路如何运行。

## 6. 对外接口分层

### 6.1 CAO 负责所有公开工作流与治理入口

包括：

- 工作流发起
- 工作流历史
- 工作流链路回放
- 公司级治理查询
- Leader 状态查询
- 周期报告收集与查询
- 行政控制台页面

### 6.2 CMO 负责所有对话入口

包括：

- `/api/cmo/chat`
- `/api/promotion/chat`

`promotion` 仍可作为营销语义别名，但实现归属 `CMO`。

### 6.3 CIO 负责内部运行查询

包括：

- 运营摘要
- 成本记录
- 审核记录
- 存储状态
- 事件、日志、追踪、持久化读写

## 7. 目录落位规则

后续判断一个文件应该放哪里，按这个顺序：

1. 它是在治理系统，还是在执行业务
2. 它是在对外暴露入口，还是在后台支撑
3. 它是在处理流程，还是在处理数据
4. 它属于哪个部门的职责闭环

具体规则：

- 对外工作流入口、治理网关、公开控制台 -> `CAO`
- 对话、进度播报、结果汇报 -> `CMO`
- 数据访问、事件、日志、观测、持久化 -> `CIO`
- 公司级控制平面、组织治理、流程编排规则 -> `CEO`
- 主链路具体动作 -> 各业务部门自己的 `skills / services / pipelines / use_cases`

## 8. 物理目录与语义目录

当前应用启动入口仍位于 `src/app/CEO/app.py`，这是技术上的应用装配位置，不代表 CEO 仍然对外承接业务。

语义归属应以路由前缀、用例归属、服务归属为准，而不是以某个启动文件的物理路径为准。

因此当前的正确理解是：

- `app.py` 是应用装配点
- `CAO / CMO / CIO` 是前台对外部门
- `CEO` 是后台治理部门

## 9. 结论

当前结构的最终判断是：

- 现有部门已经足够，不新增新的路由或生产部门
- `COO / CSO / CCO` 必须独立保留
- `CEO` 必须退居后台，只保留治理与编排内核
- 所有外部接口工作统一委托给 `CAO / CMO / CIO`

后续任何新增代码，只要还让 CEO 直接面对用户，就说明结构又开始偏了，需要立刻纠正。
