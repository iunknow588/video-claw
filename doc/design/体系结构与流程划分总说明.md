# 体系结构与流程划分总说明

## 1. 目标

本项目只保留以下八个顶层业务结构：

- `app/CEO`
- `app/CMO`
- `app/CIO`
- `app/CTO`
- `app/CFO`
- `app/CHO`
- `app/CAO`
- `app/CQO`

其他代码不再作为平级体系存在，必须并入上述结构化模块中，形成单一、清晰、一致的业务组织方式。

## 2. 总体原则

1. 顶层只保留一套组织结构，不再保留并行命名体系。
2. 每个模块只承担自己的职责边界，不跨域堆积功能。
3. 入口、服务、技能、数据、编排统一归属到对应业务模块。
4. `__init__.py` 只做最小导出，不承载业务实现。
5. 所有新代码必须按本说明落位，禁止再新增旁路目录。

## 3. 八大模块职责

### 3.1 CEO

负责全局治理与总控，包括：

- 系统目标与策略决策
- 工作流规划与调度
- 部门编排与状态观测
- 组织结构变更与优化命令
- 全局报告与控制面入口

建议承接内容：

- CEO 控制台
- CEO 工作流
- CEO 领导层配置与查询
- CEO 职责相关 API

### 3.2 CMO

负责对外沟通与展示，包括：

- 用户沟通
- 进度展示
- 报告展示
- 推广与可视化交互

建议承接内容：

- 聊天界面
- 进度面板
- 报告面板
- 推广相关服务

### 3.3 CIO

负责信息中枢与持久化能力，包括：

- 信息存储
- 日志记录
- 知识资产管理
- 查询与追踪
- 系统稳定性相关基础能力

建议承接内容：

- 存储、检索、日志、查询日志、知识库技能
- 信息事件、资源记录、知识资产模型
- 与工作流记录相关的基础服务

### 3.4 CTO

负责技术方案与提示词体系，包括：

- 提示词包设计
- 标题候选生成
- 提示词校验
- 提示词版本管理

建议承接内容：

- 提示词相关技能
- 技术策划与版本控制相关能力

### 3.5 CFO

负责财务闸门与成本控制，包括：

- 成本预估
- 余额校验
- 扣费与留痕
- 预算控制

建议承接内容：

- 估算、验资、扣费技能
- 财务记录与成本服务

### 3.6 CHO

负责人力与公共能力治理，包括：

- 公共 Agent 注册
- 能力目录治理
- 健康状态汇总

建议承接内容：

- Agent 能力、注册、健康相关技能
- 公共能力展示和维护

### 3.7 CAO

负责发布交付与平台适配，包括：

- 平台适配
- 发布执行
- 回调处理
- 发布历史
- 重试恢复

建议承接内容：

- 发布计划、发布执行、平台适配、发布回调、发布历史技能
- 对外交付链路

### 3.8 CQO

负责质量门禁与交付审核，包括：

- 内容合规
- 画面与渲染质量
- 技术规格检查
- 交付资产检查
- 质量报告

建议承接内容：

- 各类 QA 检查技能
- 质量评审与回流机制

## 4. 流程划分

### 4.1 主业务流程

默认主流程应保持以下顺序：

```text
CFO
-> Research / Research-like discovery
-> Analysis
-> CTO
-> Production
-> CQO
-> CAO
```

说明：

- CFO 负责放行与预算控制
- 发现、分析、技术整理、生产、质检、发布必须形成连续链路
- 任何步骤失败，都应在对应模块内完成回收、记录和可追踪反馈

### 4.2 控制流程

控制流程由 CEO 统一管理，负责：

- 接收任务
- 组织部门协作
- 监控工作流状态
- 触发优化与调整
- 输出总览和汇报

### 4.3 信息流程

CIO 负责所有过程数据的沉淀与查询，要求：

- 所有关键步骤都有记录
- 运行轨迹可回放
- 事件、资产、日志可以被统一查询

### 4.4 展示流程

CMO 负责面向用户和外部观察者的展示层，要求：

- 只展示，不替代核心业务判断
- 所有展示数据来自正式业务模块
- 不在展示层实现业务分叉逻辑

## 5. 代码落位规则

### 5.1 允许的归属方式

新增或迁移代码时，必须先判断它属于哪个顶层模块，再放入对应目录下。

示例：

- `app/CEO/...` 放总控、编排、领导配置
- `app/CIO/...` 放日志、检索、存储、知识
- `app/CFO/...` 放预算、扣费、验证
- `app/CQO/...` 放质检与验收

### 5.2 不允许的情况

- 新增平级业务目录
- 在 `app/skills` 下继续扩展独立业务体系
- 在 `app/services` 下继续堆积跨域业务实现
- 让 `__init__.py` 承担复杂逻辑
- 让一个功能在多个顶层目录重复出现

## 6. 目录收敛目标

最终目标是让代码结构形成以下样式：

```text
app/
  CEO/
  CMO/
  CIO/
  CTO/
  CFO/
  CHO/
  CAO/
  CQO/
```

其余能力必须并入这些模块内部，以内部子目录或内部服务单元的方式组织。

## 7. 重构顺序建议

1. 先完成文档定版，锁定职责边界。
2. 再按职责边界迁移代码入口。
3. 然后收拢技能、服务、模型、API 的归属。
4. 最后清理无效平级目录和重复导出。

## 8. 验收标准

当且仅当满足以下条件时，结构才算达标：

- 顶层只保留八个业务模块
- 不再存在并行业务命名体系
- 每个模块职责单一且可解释
- 主流程、控制流程、展示流程、信息流程均有明确归属
- 新增功能可以直接判断落点，不需要猜目录
