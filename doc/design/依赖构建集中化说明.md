# 依赖构建集中化说明

## 目标

在现有 `CEO / CMO / CIO / CTO / CFO / CHO / CAO / CQO / COO / CSO / CCO` 部门结构下，进一步消除业务服务直接读取全局配置、直接拼装外部依赖的问题，形成清晰的一致规则：

1. `services` 负责用例编排与业务流程。
2. `skills` 负责单一动作，不直接承担复杂依赖拼装。
3. 配置源统一由 `CIO` 配置平台提供。
4. 技术依赖由对应职能部门集中构建：
   - `CTO` 负责 AI Provider Client 构建。
   - `CIO` 负责存储、媒体路径、公共资源定位。

## 现状问题

优化前存在以下结构性问题：

1. `CCO / COO` 多个服务各自读取 `DEEPSEEK_* / GLM_* / SEEDANCE_*`，重复拼接 client。
2. 占位模式 `AI_USE_PLACEHOLDER_WHEN_UNCONFIGURED` 在多个服务中散落判断。
3. `MEDIA_ROOT / MEDIA_URL_PREFIX / MEDIA_BASE_URL / VIDEO_STORAGE_BACKEND` 被 `CIO / COO / CSO / CQO / CEO` 多处直接读取。
4. 媒体路径拼接、资产存在性判断、公开 URL 生成逻辑分散，不利于后续替换存储后端。

这会导致三个后果：

1. 同类依赖配置修改需要跨多个部门同步。
2. 服务层职责变厚，编排逻辑与依赖构建耦合。
3. 测试只能依赖大量 monkeypatch 全局 settings，边界不清晰。

## 调整原则

### 1. 配置归 CIO，技术装配归职能部门

- `CIO/services/config_platform/` 继续作为唯一配置装载入口。
- `CTO/services/ai_clients/` 负责把当前配置解析成可用的 AI 运行时快照与 client。
- `CIO/services/storage/` 负责把当前配置解析成存储运行时、媒体路径规则、存储 backend。

### 2. 服务只消费运行时对象或工厂函数

业务服务不再自行拼：

- `api_key`
- `base_url`
- `model`
- `timeout`
- `retries`
- `media_root`
- `media_url_prefix`
- `storage backend`

服务层只做：

1. 调用依赖工厂。
2. 组织 prompt / payload。
3. 处理业务结果。
4. 写审计、事件、仓储。

### 3. 保留单一入口，避免兼容层扩散

本项目是第一版，不引入历史兼容包装层。统一入口直接替代旧的散落读取方式：

- AI 统一走 `CTO/services/ai_clients/`
- 存储与媒体路径统一走 `CIO/services/storage/`

## 目标结构

```text
app/
  CIO/
    services/
      config_platform/        # 配置装载与校验
      storage/                # 存储运行时、媒体路径、后端工厂

  CTO/
    services/
      ai_clients/             # AI provider runtime + client factory

  CCO/
    services/
      content_creation/       # 只消费 DeepSeek runtime/client

  COO/
    services/
      script_management/      # 只消费 GLM runtime/client
      video_production/       # 只消费 Seedance runtime/client + CIO storage factory
      asset_management/
      composition/

  CQO/
    skills/
      render_output_check/    # 只消费 CIO 媒体资产判断能力
```

## 运行时职责划分

### CTO: AI Provider Runtime

`CTO/services/ai_clients/` 统一提供：

1. Provider 配置快照
2. Placeholder 策略判断
3. Client 构造函数
4. HTTP timeout / retry 统一策略

典型接口：

```python
provider = get_ai_provider_config("glm")
client = build_glm_client(provider)
```

服务层只保留：

```python
self.provider = get_ai_provider_config("glm")
self.client = build_glm_client(self.provider)
```

### CIO: Storage Runtime

`CIO/services/storage/` 统一提供：

1. 当前存储后端快照
2. 本地媒体目录解析
3. 公开媒体 URL 生成
4. 资产存在性判断
5. 视频存储 backend factory

典型接口：

```python
runtime = get_storage_runtime()
path = runtime.resolve_path("subtitles", "trace-1.srt")
url = runtime.build_public_url("videos/task-1.mp4")
```

## 业务边界结果

调整后各部门边界更清晰：

- `CCO` 只关心分析逻辑，不关心 DeepSeek 参数装配。
- `COO` 只关心脚本、视频、字幕、旁白、渲染流程，不关心媒体根目录规则。
- `CQO` 只做质检规则，不自己解析媒体目录。
- `CEO` 作为 API 与 orchestration 入口，不再扩散基础设施判断。
- `CIO` 成为配置与资源定位中心。
- `CTO` 成为外部 AI 技术接入中心。

## 收益

1. **清晰**：依赖构建位置固定，不再跨部门散落。
2. **简单**：服务只写业务逻辑，不重复写配置读取。
3. **一致**：同类 provider / 存储后端统一入口、统一策略。
4. **可测试**：测试只需替换工厂输出或运行时快照。
5. **可扩展**：后续新增 provider / storage backend，只需在集中模块扩展。
