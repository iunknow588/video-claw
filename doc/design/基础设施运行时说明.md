# 基础设施运行时说明

## 目标

继续把底层依赖从全局平铺配置读取中收拢，明确基础设施归属：

1. `CIO` 持有数据库与 Redis 运行时。
2. 中立应用装配层持有应用启动入口，`CEO` 只持有应用运行时配置与日志配置。

## 问题

优化前，以下入口各自直接读取全局配置：

1. `CIO/db/session/`
2. `CHO/tasks/celery_tasks/`
3. `src/app/app.py`
4. `CEO/core/logging/`
5. `main.py`

这导致两个问题：

1. 同一类基础设施参数在多个入口重复解释。
2. 模块归属虽然已经按部门拆开，但运行时边界还不够明确。

## 方案

### CIO 基础设施运行时

在 `CIO/services/` 下提供：

```text
database_runtime/   # 数据库连接、连接池、版本快照
redis_runtime/      # Redis / Celery broker / backend
```

职责：

1. 读取 `cio_database`
2. 读取 `cio_redis`
3. 构造统一连接参数
4. 向 `session` 与 `celery` 提供稳定入口

### CEO 应用运行时配置

在 `CEO/services/` 下提供：

```text
application_runtime/  # 应用身份、server、logging
```

职责：

1. 读取 `ceo_application`
2. 提供应用身份与 server 参数
3. 提供日志参数
4. 作为 `src/app/app.py` 与 `main.py` 的统一运行时口径

## 调整后的边界

### CIO

- 数据库 URL、连接池、版本感知
- Redis host/port/db/password
- Celery broker/backend URL

### 中立装配层 + CEO

- `src/app/app.py` 负责应用创建、生命周期与挂载
- `CEO` 提供 App 名称与版本
- debug / env
- HTTP server host / port / workers
- logging level / format / file

## 结果

收敛后形成以下结构：

1. `session` 不再自己解释数据库配置。
2. `celery` 不再自己解释 Redis 配置。
3. `src/app/app.py / main.py` 不再自己解释应用启动参数。
4. `logging` 不再直接依赖平铺日志字段。

这样后续如果继续扩展：

1. 多实例部署参数
2. 不同环境的日志策略
3. 多 Redis 角色拆分
4. 数据库只读/写分离

都可以在 runtime 层扩展，而不用修改业务入口。
