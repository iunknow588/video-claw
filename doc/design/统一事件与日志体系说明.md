# 统一事件与日志体系说明

## 1. 目标

本次调整要解决一个核心问题：

`WorkflowExecutionRecorder` 之前同时承担了三类职责：

- 步骤记录
- 事件发射
- CIO 产物持久化

这使它既像记录器，又像事件总线，又像 CIO 仓储入口，职责过重，测试和扩展都不清晰。

本次重构后，事件与日志体系拆成两层：

- `CIO/services/event_bus`
- `CIO/services/observability`

## 2. 新结构

```text
departments/CIO/services/
  event_bus/
    __init__.py
    event_publisher.py
    event_subscriber.py
    event_store.py
  observability/
    __init__.py
    trace_collector.py
    metrics_reporter.py
    log_aggregator.py
```

## 3. 责任划分

### 3.1 event_bus

负责“事件被发布以后发生什么”。

- `event_publisher`
  - 负责把事件发给订阅者
  - 负责把对外 `event_callback` 作为一种发布目标统一处理

- `event_subscriber`
  - 定义订阅者对象
  - 为后续增加审计、告警、通知等订阅行为留出边界

- `event_store`
  - 负责事件持久化
  - 统一把事件写入 CIO 的信息事件表
  - 对产物类事件额外写入 artifact 记录

### 3.2 observability

负责“事件如何被追踪和汇总”。

- `trace_collector`
  - 只负责链路追踪
  - 将 trace 类事件写入 `workflow_steps`

- `metrics_reporter`
  - 负责轻量指标累计
  - 为后续接入监控系统预留统一位置

- `log_aggregator`
  - 负责把统一事件转换成结构化日志视图
  - 统一决定 level / message / context 的组织方式

## 4. Recorder 新职责

Recorder 被收缩为单一编排入口：

```python
class WorkflowRecorder:
    async def record(self, event: WorkflowEvent):
        await self.event_bus.publish(event)
        await self.trace_collector.append(event)
```

在此基础上，它只保留两类辅助动作：

- 把 skill 调用包装成标准事件
- 把阶段状态、产物持久化、内部日志转换成标准事件后交给上面两层处理

也就是说，Recorder 现在只“组织记录动作”，不再直接拥有 CIO 仓储实现细节。

## 5. 运行流向

### 5.1 skill 调用事件

`call_skill()` 产生：

- start
- finish
- fail

这些事件会：

1. 进入 `event_bus`
2. 写入 `information_events`
3. 被 `trace_collector` 追加到 `workflow_steps`
4. 被 `metrics_reporter` 计入轻量指标

### 5.2 阶段状态事件

工作流阶段开始/成功/失败时，产生 status 事件。

这些事件会：

1. 进入 `event_bus`
2. 通过 `event_publisher` 发给外部回调
3. 写入统一事件日志

### 5.3 产物事件

像 `finance.bundle`、`research.bundle`、`qa.bundle` 这类产物，不再由 Recorder 直接调用 `lead.cio.store`。

现在改为：

1. 生成 artifact 事件
2. `event_store` 识别并写入 `artifact_records`
3. 同时保留统一信息事件记录

## 6. 结构收益

这次拆分后的收益非常明确：

1. Recorder 不再混合三重职责
2. CIO 成为真正的事件与观测承载部门
3. 产物持久化不再伪装成一次 skill 调用
4. 后续增加订阅者、告警、指标导出时，不需要继续改 CEO 编排层

## 7. 当前边界结论

现在这条边界是清晰的：

- CEO 负责编排
- CIO 负责事件总线、日志、追踪、指标、产物记录
- Recorder 只是把工作流动作转成标准事件并交给 CIO 体系处理

这比之前把记录、发射、持久化全部塞进一个 Recorder 里更容易理解，也更方便持续扩展。
