# 配置与依赖管理集中化说明

## 1. 调整目标

本次调整的目标，不是再给 `Settings` 类继续打补丁，而是把配置体系改造成一套清晰、单轨、可验证、可按部门治理的结构。

调整后遵循五条规则：

- 只保留一套配置来源，不再并存 `config.yaml + BaseSettings`
- 配置按部门与基础设施拆分，不再堆进一个超大类
- 环境变量只在配置读取阶段解析，不再散落在业务代码里
- 配置验证分为类型校验与业务规则校验两层
- 依赖实例要感知配置版本，避免“配置更新了，依赖还拿旧值”

## 2. 组织归属

配置管理按职责拆分如下：

- `CEO`：治理规则、全局应用标识、权限边界
- `CIO`：配置平台、读取/验证/缓存、基础设施配置、依赖构建
- `CFO`：预算、成本阈值、定价
- `COO`：生产参数、渲染/合成默认值
- `CSO`：热点采集策略

当前不需要新增专门的“配置部门”。

原因很简单：

- 配置平台能力本身属于 `CIO`
- 谁拥有业务，谁定义自己的配置 Schema
- 谁拥有治理权限，谁定义修改规则

## 3. 调整前的问题

当前配置体系存在四个根问题：

1. `config.yaml` 与 `app/CEO/core/config/__init__.py` 双轨并存
2. `Settings` 直接读环境变量，测试和热更新都很别扭
3. 业务配置、基础设施配置、治理配置全部揉在一起
4. `db/session` 这类依赖在模块导入时就固化配置，后续无法随配置刷新

## 5. 配置分层

### 5.1 governance

`config/governance/` 只放 CEO 级治理与全局应用配置：

- `application.yaml`：应用名、版本、debug、server、logging、monitoring
- `permissions.yaml`：谁能修改哪个配置域
- `departments.yaml`：配置域与部门归属
- `workflow.yaml`：保留工作流治理入口

### 5.2 infrastructure

`config/infrastructure/` 只放 CIO 基础设施配置：

- `database.yaml`
- `redis.yaml`
- `ai_providers.yaml`
- `storage.yaml`

### 5.3 departments

`config/departments/` 只放业务部门自有规则：

- `CFO/finance.yaml`
- `COO/production.yaml`
- `CSO/hotspot.yaml`

## 6. 平台职责

### 6.1 ConfigReadSkill

- 读取 YAML / JSON
- 递归解析 `${ENV_VAR}` 占位符
- 不做业务判断

### 6.2 ConfigValidateSkill

- 根据目标 Pydantic 模型做校验
- 捕获并结构化返回错误
- 承接业务规则校验结果

### 6.3 ConfigTransformSkill

- 把已验证的原始 dict 转成强类型模型

### 6.4 ConfigManager

`ConfigManager` 是真正的协调者，负责：

- 配置发现
- 缓存
- 按域加载
- 失效与刷新
- 暴露配置版本号，供依赖判断是否需要重建

## 7. 依赖管理策略

“依赖管理集中化”这次不再理解成再造一个大容器，而是把依赖构建和配置版本绑定起来。

### 7.1 数据库依赖

数据库引擎与 `sessionmaker` 改为懒构建：

- 首次访问时读取 `cio_database`
- 若配置域版本变化，则重建 engine / session factory
- 不再在模块导入时把配置固化死

### 7.2 其他依赖

- AI Client：服务实例化时读取当前配置
- Storage：每次通过工厂读取当前配置
- Logging：启动时按治理配置初始化

这意味着配置更新后，至少新增实例与懒依赖会看到新值，不会继续被旧的 import-time 常量绑住。

## 8. 对外接口

`app/CEO/core/config` 继续作为唯一配置访问入口，但内部不再是 `BaseSettings`。

它的职责变成：

- 持有全局 `settings` 运行时对象
- 汇总各配置域
- 提供旧调用点所需的扁平属性
- 提供 `reload()`、`reload_domain()`、`version()` 这类运行时能力

这里保留的是“统一入口”，不是“兼容层双实现”。

## 9. 本次落地范围

本次代码修改范围如下：

1. 创建新的 `src/config/` 目录并拆分原 `config.yaml`
2. 创建 `CIO/services/config_platform/`
3. 为 `CEO / CIO / CFO / COO / CSO` 补齐配置 Schema 与 Agent
4. 重写 `app/CEO/core/config/__init__.py`
5. 重构 `app/CIO/db/session/__init__.py`，让数据库依赖跟随配置版本
6. 删除旧的 `src/config.yaml`

## 10. 结果预期

改完以后，配置体系会变成下面这个判断方式：

- 看目录就知道配置归谁
- 看 Schema 就知道规则在哪里
- 看 Agent 就知道部门怎么使用配置
- 看 `ConfigManager` 就知道配置如何被加载、缓存、刷新
- 看依赖模块就知道它是否能跟着配置变更一起更新

这比继续维护一个不断膨胀的 `Settings` 类清楚得多，也更符合当前 CxO 架构。
