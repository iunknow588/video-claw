# 系统模块与技能结构说明

## 1. 文档目的

这份文档聚焦 `src/app/` 当前真实存在的模块结构，说明：

1. 顶层 11 部门目录如何划分。
2. `skills / services / use_cases / models / runtime` 分别负责什么。
3. 新增代码应该优先落到哪里。

## 2. 当前顶层结构

```text
src/app/
|-- CEO/
|-- CIO/
|-- CFO/
|-- CHO/
|-- CMO/
|-- CAO/
|-- COO/
|-- CQO/
|-- CTO/
|-- CSO/
`-- CCO/
```

这 11 个目录就是当前系统正式采用的业务组织边界。

## 3. 通用部门目录形态

大多数部门目录已经逐步收敛到下面这种结构：

```text
Department/
|-- agent/
|-- leader/
|-- services/
|-- skills/
`-- config/        # 仅部分部门具备
```

说明如下：

- `agent/`：部门 Agent 的装配入口。
- `leader/`：部门 Leader 的归口定义。
- `services/`：部门级服务、编排、聚合、用例承接。
- `skills/`：原子能力、单点执行能力、轻量可复用处理单元。
- `config/`：部门自身 schema、agent 配置、参数定义。

## 4. 重点部门说明

### 4.1 CEO

`CEO` 是当前治理与编排中心：

```text
CEO/
|-- router/
|-- services/
|   |-- application_runtime/
|   |-- config_governance/
|   |-- control/
|   |-- control_plane/
|   `-- orchestration/
|-- core/
|-- leaders/
`-- skills/
```

职责重点：

- `router/`：统一 API 入口。
- `services/orchestration/`：工作流编排核心。
- `services/control_plane/`：治理控制面能力。
- `services/control/`：治理执行支撑。
- `services/config_governance/`：配置治理与规则约束。

### 4.2 CIO

`CIO` 是数据与基础设施中台：

```text
CIO/
|-- db/
|-- models/
|-- schemas/
|-- services/
|   |-- config_platform/
|   |-- data_access/
|   |-- database_runtime/
|   |-- event_bus/
|   |-- knowledge/
|   |-- leader_reports/
|   |-- observability/
|   |-- operations/
|   |-- redis_runtime/
|   |-- runtime_assets/
|   |-- storage/
|   |-- workflow_runs/
|   |-- workflow_steps/
|   `-- use_cases/
`-- skills/
```

职责重点：

- `models/`：核心持久化模型。
- `data_access/`：Repository 层。
- `runtime_assets/`：统一解析仓库根目录下 `runtime/` 路径。
- `storage/`：媒体、视频、对象存储能力。
- `leader_reports/`：Leader 汇报持久化与查询。
- `observability/`：日志、指标、追踪聚合。

### 4.3 COO

`COO` 负责生产、组合与资产处理：

```text
COO/
|-- services/
|   |-- asset_management/
|   |-- composition/
|   |-- script_management/
|   |-- video_production/
|   `-- use_cases/
`-- skills/
```

职责重点：

- `script_management/`：脚本草拟、审校相关服务。
- `video_production/`：视频生产与任务处理。
- `asset_management/`：字幕、配音、媒资管理。
- `composition/`：渲染与成片组装。

### 4.4 其他业务部门

- `CSO`
  - 热点采集、去重、排序、趋势情报、素材检索。
- `CCO`
  - 内容分析、结构提取、风险提取、可复用内容沉淀。
- `CTO`
  - prompt package、标题候选、提示词校验与版本管理。
- `CQO`
  - 质量门禁、技术规格检查、合规检查、交付前质检。
- `CAO`
  - 对外网关、发布计划、平台适配、回调、交付查询。
- `CFO`
  - 预算估算、余额校验、扣费与财务门禁。
- `CHO`
  - 公共 Agent 注册、能力目录、健康状态。
- `CMO`
  - 对话入口、进度展示、报告视图。

## 5. `skills` 与 `services` 的边界

### 5.1 `skills/`

适合承载：

- 原子动作
- 单点规则判断
- 可复用输入输出转换
- 明确单步骤执行

例如：

- 标题候选生成
- prompt 校验
- 预算校验
- 平台适配
- QA 报告生成

### 5.2 `services/`

适合承载：

- 多个 skill 的组合
- 带状态的执行过程
- 与数据库、Redis、对象存储、外部 API 交互
- 用例编排与聚合

例如：

- `CEO/services/orchestration/`
- `CIO/services/storage/`
- `COO/services/video_production/`
- `CCO/services/content_creation/`

## 6. `services/use_cases/` 的作用

`services/use_cases/` 是当前结构中很重要的一层，主要职责是：

1. 承接路由输入。
2. 调用本部门 service、repository 或 pipeline。
3. 组装返回结果。
4. 保持路由层薄、业务层清楚。

这也是“入口统一，职责下沉”的主要落地方式。

## 7. 当前不再采用的旧结构

以下写法不再代表当前结构：

- `src/app/services/*` 作为全局业务中心
- `src/app/api/*` 作为主路由宿主
- `src/app/models/*` 平铺表达全部业务归属
- `src/app/skills/*` 无边界扩散

这些旧路径只适合出现在历史草案或迁移背景说明中，不应再作为现行结构引用。

## 8. 新增代码的落位建议

新增代码时，优先按下面顺序判断：

1. 先确定归属哪个部门。
2. 如果是入口请求处理，放到该部门 `services/use_cases/`。
3. 如果是全局工作流编排，优先进入 `CEO/services/orchestration/`。
4. 如果是持久化、日志、事件、存储、运行时路径，优先进入 `CIO/services/`。
5. 如果是原子能力，进入对应部门 `skills/`。

## 9. 结论

当前模块化整理的重点，不是“服务名叫不叫 service”，而是：

1. 是否按部门闭环组织。
2. 是否让路由、用例、服务、技能、数据访问各归其位。
3. 是否避免重新引入全局平铺目录。
