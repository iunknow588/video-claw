# CEO 隐身原则与对外接口分层说明

## 1. 设计原则

在当前系统里，CEO 的定位是治理核心，而不是前台执行者。

因此需要遵守“CEO 隐身原则”：

- CEO 不直接面对用户
- CEO 不直接暴露公开 API
- CEO 不直接承载公开页面
- CEO 只通过委托链路影响系统行为

这不是命名偏好，而是结构规则。

## 2. 对外接口三分法

### 2.1 CAO：行政与流程入口

CAO 对外负责：

- 工作流提交
- 工作流查询
- 治理信息网关
- 控制台页面
- 交付结果展示

### 2.2 CMO：对话入口

CMO 对外负责：

- 用户消息接收
- 事件流返回
- 进度播报
- 结果总结

### 2.3 CIO：内部运行查询

CIO 对内负责：

- 运营摘要查询
- 成本与审核记录查询
- 存储状态查询
- 事件、日志、追踪查询

## 3. CEO 的正确调用位置

CEO 只应出现在下面的位置：

- `control`
- `control_plane`
- `orchestration`
- 被 `CAO / CMO / CIO` 的 use case 间接调用

CEO 不应出现在下面的位置：

- URL 前缀
- 页面路由
- 用户消息入口
- 公开控制台入口

## 4. 为什么这样更清晰

这样分层以后：

- 用户一眼就知道“谁在接待我”
- 运维一眼就知道“谁在给我看运行状态”
- 开发一眼就知道“谁是后台治理内核”

对外职责与内部治理职责不会再混在一起。

## 5. 判定标准

以后审查代码时，可以直接用这四个问题判断结构是否跑偏：

1. 新增的公开接口是不是又挂在了 CEO 下面
2. 用户页面是不是又出现了 `/ceo`
3. 对话入口是不是绕过了 CMO
4. 流程入口是不是绕过了 CAO

只要任一答案是“是”，就需要重新调整结构。
