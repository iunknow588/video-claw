# 代码重构落地清单

## 1. 当前目标

把代码整理到明确的 11 部门结构中，并确保每项能力只有一个真实归属，不再混用旧的平铺式 `services` 结构。

当前正式采用的顶层部门为：

- `departments/CEO`
- `departments/CIO`
- `departments/CFO`
- `departments/CHO`
- `departments/CMO`
- `departments/CAO`
- `departments/COO`
- `departments/CQO`
- `departments/CTO`
- `departments/CSO`
- `departments/CCO`

## 2. 重构原则

1. 先定职责，再定目录。
2. 一项能力只保留一个主入口。
3. `services` 负责编排、聚合、用例承接。
4. `skills` 负责原子动作或可复用执行单元。
5. 不继续扩散旧的平铺路径别名。
6. 对外接口不再以 CEO 作为公开门面前缀。

## 3. 结构收口要求

### 3.1 CEO

保留以下职责：

- `services/orchestration/`
- `services/control/`
- `services/control_plane/`
- `services/config_governance/`

不再把 CEO 当成外部业务 API 门面层。

### 3.2 CAO

承接对外访问与交付面：

- 工作流网关
- 治理查询网关
- 发布交付视图
- 公共控制台

### 3.3 CIO

统一基础设施与数据面：

- `models/`
- `db/`
- `services/data_access/`
- `services/storage/`
- `services/runtime_assets/`
- `services/leader_reports/`
- `services/observability/`
- `services/event_bus/`

### 3.4 业务部门

`CSO / CCO / CTO / COO / CQO / CFO / CHO / CMO` 各自保留本部门的：

- `skills/`
- `services/`
- `services/use_cases/`
- `config/`
- `leader/`
- `agent/`

## 4. 入口与路径约束

当前仓库入口应固定为：

- 根目录 `Dockerfile`
- 根目录 `docker-compose.yml`
- 根目录 `alembic.ini`
- 根目录 `pytest.ini`
- 应用入口 `src/main.py`
- 应用装配入口 `src/app/app.py`
- API 装配入口 `src/app/router/`

运行期资产统一进入：

- `runtime/logs/`
- `runtime/media/`

不再把日志、产物、临时数据库混放在 `src/` 下。

## 5. 命名整理方向

需要持续清理两类旧命名：

1. 公开接口或 DTO 中把 `CEO` 当成用户侧角色前缀的命名。
2. 指向旧平铺结构的描述，例如 `src/app/services/*`、`src/app/models/*`、`src/app/api/v1/*`。

## 6. 文件清理原则

可以直接清理或迁移的内容：

- 运行期日志
- 媒体产物
- 临时缓存
- 测试缓存
- 只用于过渡的旧说明文件

需要保留但应单独归档说明的内容：

- 历史设计草案
- 联调记录
- 优化报告
- 结构整改提案

## 7. 验收标准

重构完成后，应满足：

1. 用户从外部入口看不到 CEO 直接充当公开业务前台。
2. `src/app/` 只保留中立应用装配代码，不再承载部门语义。
3. `src/departments/` 成为唯一的部门职责目录。
4. `runtime/` 成为唯一运行期资产落点。
5. 根目录成为统一启动、测试、迁移入口。
6. 文档中的正式路径与源码实际路径一致。
